# RTrace 1.0.0.9000

## Bug fixes

* `aggregate_scores()` raised `subscript out of bounds` for any module id
  missing from a partial `weights` vector instead of falling back to the
  documented default weight of `1.0` (#6).
* `reproducibility.externalDownload` flagged `readLines()`, an ordinary
  local-file-reading function, as an external network download (#7).
* `datatrace.readError` treated any `read.table()` warning -- including the
  extremely common "incomplete final line found" for a file with no
  trailing newline -- as an unparseable file, also spuriously tripping
  `datatrace.encodingIssue` (#8).
* `packageqa.testCoverage` counted the boilerplate `tests/testthat.R`
  runner stub (generated by `usethis::use_testthat()`) as a real test file,
  so a package with zero actual tests could still pass the check (#9).
* `docstrace.examplesQuality` used a `[^}]*}` regex to find the end of an
  `\examples{}` block, which stopped at the first nested closing brace and
  misclassified substantive examples as trivial (#10).
* `reproducibility.*`, `datatrace.*`, `docstrace.*`, and `packageqa.*` rule
  types could not be referenced in `rtrace.yml` at all -- `validate_config()`
  rejected them as unknown, and even past that, the domain engines ignored
  `config` entirely and always ran every rule at its hardcoded default
  severity/params. These 37 rules are now configurable the same way as
  every other rule: disable one with `enabled: false`, or override its
  `severity`/params, without affecting the rest (#11).
* `discover_plugins()` ran `requireNamespace()` -- and hence executed
  `.onLoad()` -- for *any* installed package that merely set
  `Config/rtrace/plugin: true` in its DESCRIPTION, with no validation that
  the package was trusted. It now only loads packages the caller has
  explicitly allowlisted via the new `plugin_allowlist()` (the
  `rtrace.plugin.allowlist` option or `RTRACE_PLUGIN_ALLOWLIST` env var);
  self-declaration alone is no longer sufficient (#13).
* Documentation for plugin discovery claimed it runs "at package load
  time"; it has never been called from `.onLoad()`. Docs now describe it
  accurately as on-demand only (#20).
* The bug report issue template told reporters to run
  `rtrace::rtrace_sitrep()`, a function that does not exist in any
  namespace. It now asks for `sessionInfo()` (#21). The same
  lowercase-package-name mistake (`rtrace::` instead of `RTrace::`) was
  also present in `SECURITY.md`, an ADR, and two `man/` pages; all are
  corrected.
* `README.md`, `DESCRIPTION`, generated SARIF output (`informationUri`),
  and several other files pointed at the placeholder org `rtrace-dev/rtrace`
  and domain `@rtrace.dev`, none of which resolve. Replaced throughout with
  the real repository (`RonaldRonnie/Rtrace`) and maintainer contact (#15,
  #16).

# RTrace 1.0.0

First stable release. Adds the Trace Platform layer on top of the 0.1.0
architecture-governance core: four additional domain engines, a unified
scoring system, a REST API, new CLI commands, and a plugin system.

## New features

* Four new platform modules, each with its own scoring model and CLI
  command, registered alongside `rtrace` via `register_module()`:
  * **Reproducibility** (`reproducibility.*`, 8 rules) -- renv.lock
    presence, missing `set.seed()`, external downloads, temp-file cleanup,
    undocumented env vars, portable paths, reproducible report chunk
    options.
  * **DataTrace** (`datatrace.*`, 13 rules) -- CSV/TSV/JSON parse errors,
    missing headers, encoding issues, schema documentation, FAIR
    (Findable/Accessible/Interoperable/Reusable) compliance checks,
    missing values, duplicate rows.
  * **DocsTrace** (`docstrace.*`, 8 rules) -- README presence and quality,
    vignette coverage, pkgdown config, example quality, changelog,
    contributing guide, citation file.
  * **PackageQA** (`packageqa.*`, 8 rules) -- DESCRIPTION completeness,
    NAMESPACE hygiene, test coverage, license, version format, NEWS
    format, maintainer contact.
* `platform_scan()`: runs every registered module against a project root
  in one call and returns a `trace_platform_result` with per-module scores
  and a combined diagnostic set.
* `aggregate_scores()`: weighted-mean aggregation of per-module
  `trace_score`s into a single platform score.
* `reporter_dashboard()`: HTML dashboard combining every module's score,
  the architecture layer diagram, and full diagnostic listing.
* Optional REST API (`start_api()` / `build_api_router()`, requires the
  `plumber` package): `/health`, `/scan`, `/scan/full`, `/rules`,
  `/modules`, `/report/html`, with bearer-token authentication and a
  project-root allowlist enforced on every path-accepting endpoint.
* Seven new CLI commands: `platform-scan`, `datatrace`, `docstrace`,
  `pkgqa`, `health`, `reproducibility`, `api`.
* Plugin discovery system (`discover_plugins()`, `list_plugin_packages()`,
  `is_rtrace_plugin()`): installed packages can self-register rules via a
  `Config/rtrace/plugin: true` DESCRIPTION field.

## Known limitations

* `trace-cloud/` (a hosted-SaaS scaffold: API, worker, and web frontend)
  shipped experimentally in this release and has since been moved to a
  standalone repository; it is no longer part of this package.

# RTrace 0.1.0.9000

## New features

* Initial scaffold of RTrace: configuration loader, project scanner, base-R
  parser wrapper, dependency graph builder, extensible rule engine,
  diagnostics model, and console/JSON/Markdown reporters.
* Built-in rules: `structure.requiredDirs`, `dependency.forbidden`,
  `dependency.circular`, `complexity.cyclomatic`, `complexity.functionLength`,
  `complexity.fileLength`, `antipattern.globalAssign`, `antipattern.assign`,
  `antipattern.setwd`, `antipattern.hardcodedPath`, `documentation.missing`.
* CLI commands: `scan`, `init`, `validate`, `list-rules`, `describe-rule`,
  `config`, `version`, `help`.
* SARIF 2.1.0 reporter (`--format sarif`) for GitHub code-scanning upload.
* `testing.missingTests` rule: flags functions never referenced under
  `tests/` (disabled by default; complements, not replaces, `covr`).
* `package.deprecatedApi` rule: flags calls to project-configured
  deprecated functions, bare or namespace-qualified.
* HTML reporter (`--format html`): standalone, dependency-free report with
  inline CSS and escaped diagnostic content.
* CSV reporter (`--format csv`) and XML reporter (`--format xml`, requires
  the `xml2` package).
* `ecosystem.shinyStructure` rule: flags conflicting (`app.R` plus
  `ui.R`/`server.R`) or missing Shiny entrypoints. Self-gated on the
  project actually importing `shiny`, so it's enabled by default.
* Incremental scanning: an opt-in (`--cache` / `use_cache = TRUE`)
  content-hash-keyed AST parse cache (`.rtrace_cache/ast-cache.rds`).
  Diagnostics are always recomputed for the full project, so results are
  identical with or without caching — see ADR 0003.
* `ecosystem.targetsStructure` rule: flags `targets` usage with no
  `_targets.R` at the project root.
* `ecosystem.plumberStructure` rule: flags `plumber` usage with no `#*`
  route annotations anywhere in the project.
* `rtrace doctor` CLI command: environment and project setup diagnostics
  (R/RTrace versions, suggested-package availability, `rtrace.yml`
  validity, RStudio Project detection, AST cache state) without running a
  scan.
* The HTML reporter now renders an "Architecture Overview" section: an
  inline SVG diagram of the configured `layers:` dependency graph, with
  cyclic edges drawn in red. Pure SVG, no external JS/CSS — still a single
  standalone file. `reporter_html()` gained optional `layers`/
  `layer_graph` parameters (default empty, so its primary `diagnostics`-only
  contract is unchanged).
* `rtrace benchmark` CLI command: times each scan phase (file walk,
  parsing, dependency graph construction) and each enabled rule's
  evaluation, printing a slowest-first breakdown. Supports `--cache`.
* RStudio Addin ("RTrace: Scan Project"): scans the active project and
  opens an HTML report (including the architecture diagram) in the
  Viewer pane. `rstudioapi` added to `Suggests`.

## Bug fixes

* The Trace Platform REST API (`start_api()`/`build_api_router()`, `rtrace
  api`) had no authentication and no path restriction: any caller could
  scan or generate an HTML report for an arbitrary filesystem path (e.g.
  `/etc`), and `start_api()` accepted any `host` including `0.0.0.0` with
  no way to require a credential. Every endpoint now enforces an optional
  bearer token (`token=`/`RTRACE_API_TOKEN`, `Authorization: Bearer`) and
  a mandatory allowlist (`allowed_roots=`/`--allowed-root`, default the
  working directory) that request `root` paths must resolve within;
  `start_api()` refuses to bind a non-loopback host unless a token is
  configured (#4).
